The Museum Companion

Project Overview

The Museum Planning Companion is a digital tool designed to bridge the gap between educational goals and logistical reality. By providing a centralized planning space, it transforms the field trip experience for educators, allowing them to focus on learning rather than logistics.

The Problem

Tara, an elementary school teacher managing groups of 20+ students, struggles to maintain group safety and cohesion because she lacks a centralized tool to calculate required chaperone-to-student ratios and obtain a clear, accessibility-aware guide for movement, dining, and seating within the museum.

The initial launch layout established the core user flow from logistics calculation to floor plan generation, prioritizing quick task completion for educators.

Systems Optimization & Defensive Design

As a product designer with a Computer Engineering background, I don't design for static canvases—I design for dynamic, unpredictable backend data. Reviewing Iteration 0 revealed critical production vulnerabilities that required programmatic and structural layout overhauls.

Phase 0: The Baseline MVP

The Truncation Risk

  • The Layout: A 2x2 grid layout for upcoming schedules.
  • The Failure: Forcing a linear sequence into a grid increased cognitive load. More importantly, it failed defensive design parameters: variable server strings like "Museum of Civilization" were truncated into "Museum of Civilizat...", breaking data integrity.
Phase 1: The Heuristic Redesign

Data Density Overhaul

  • The Layout: Transformed the grid into a linear vertical timeline. Added rich operational metrics tracking consent slips ("21/28") and color-coded status pills.
  • The Critique: While it solved text truncation and provided high-utility metrics, a deep architectural review exposed information redundancy—the active upcoming trip was duplicated in the pipeline list below, cluttering screen space.
Phase 2: The Production-Ready State

Logic-Filtered UI

  • The Layout: Implemented backend query logic rules directly into the interface architecture.
  • The Solution: The primary header card retains the absolute next active trip requiring immediate action. The 'Upcoming Trips' list dynamically filters out the active ID, displaying only the subsequent pipeline. This eliminated redundant components, optimized white space, and preserved dynamic data flexibility.
